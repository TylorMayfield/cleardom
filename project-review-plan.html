ClearDOM Project Review Plan

Review date: July 8, 2026. ClearDOM is in a strong early-product shape: the core scanner, CLI, runtime checks, rules, fixtures, and release packaging all build and test cleanly. The next work should focus on reducing adoption friction and tightening the product edges that can confuse a first-time user.

What Is Working

Strong

Core scanner architecture is coherent

src/scanner.ts has a clear flow: resolve config, collect files, parse semantic or lightweight source, run rules, add runtime findings, apply suppressions and baseline, score, then format. That is the right backbone for a CLI scanner.

Strong

Framework coverage is unusually broad

Adapters cover JSX/TSX, HTML, Vue, Svelte, Astro, Angular templates, and MDX. Fixtures also exercise Next.js, React Native, monorepos, framework matrices, and design-system mappings.

Strong

The adoption story is thoughtful

init, doctor, baselines, suppressions, PR review, SARIF, GitHub inline comments, and fix prompts make this feel like a product instead of just a rules library.

Strong

Tests cover real product behavior

The suite exercises CLI commands, source adapters, runtime interactions, rule behavior, GitHub review helpers, standards selection, benchmarks, and fixtures. This is a good safety net for continued iteration.

Strong

WCAG positioning is credible

The benchmark fixture and WCAG documentation give the project a defensible standards story. The split between automated, needs-review, and manual-guidance rules is exactly the right honesty.

Strong

Runtime checks add real differentiation

Chromium-backed checks for contrast, focus visibility, target size, reflow, skip links, text spacing, hover/focus content, keyboard traps, and focus obstruction cover issues static source scans cannot see.

What Is Not Working Yet

High impact

Default project scans are too noisy in this repo

node dist/cli.js scan . --format json found 390 findings and a 5/100 score because it scans intentional broken examples and generated benchmark reports. node dist/cli.js scan src is clean. The product needs a root config or documented dev command that separates source health from fixture health.

Maintainability

CLI responsibilities are concentrating

src/cli.ts is about 1,112 lines and owns command routing, argument parsing, config initialization, install flows, baseline commands, browser commands, native commands, and help text. It works, but adding commands will keep getting more brittle.

Maintainability

HTML report rendering is duplicated

src/format.ts and src/report.ts both implement similar ClearDOM scan report HTML. That invites drift between scan --format html and report --format html.

Product

First-run guidance can still overwhelm

The README is comprehensive, but it presents a lot of commands and configuration at once. A first-time user needs a smaller happy path: local scan, changed-file scan, baseline, PR review, runtime checks.

Signal quality

Manual-guidance rules need careful defaults

The catalog includes many AAA and contextual risk rules. That breadth is useful, but if enabled too eagerly it can make ClearDOM feel noisy. The UI and CLI should keep confidence, detection mode, and next action very visible.

Verification

Runtime tests are valuable but slow

The full suite passed, but runtime interaction tests dominate wall time. As more runtime checks land, the project will need a split between fast unit checks and slower browser integration checks.

Priority Plan

Next 1-2 Weeks

  1. Add a root cleardom.config.json or dev script that scans src by default and excludes generated benchmark reports.
  2. Split src/cli.ts command handlers into focused modules: scan, init, install, review, baseline, fix, browser, native.
  3. Unify the duplicated HTML report builder so scan --format html and report --format html share one renderer.
  4. Add pnpm test:fast and pnpm test:runtime scripts so local development does not always require the full browser suite.

Next 3-6 Weeks

  1. Improve first-run output: show the top 3 findings, explain why anything was skipped, and offer one exact next command.
  2. Make rule docs generated from the rule catalog so README anchors, explain, SARIF metadata, and docs cannot drift.
  3. Expand safe mechanical fixes for high-confidence static rules: missing decorative image alt, native roles, positive tabindex, simple missing labels.
  4. Add fixture-level expectations for intentional example failures so noisy fixtures are tested without being confused with repo health.

Next 6-10 Weeks

  1. Ship a clearer report experience: grouped findings by route/file/rule, severity filters, confidence filters, and copyable remediation steps.
  2. Add more runtime route ergonomics: clearer crawl diagnostics, route budgets, auth failure hints, and Storybook story grouping.
  3. Review component preset semantics, especially whether placeholder-like props should count as names for design-system text inputs.
  4. Publish a small public benchmark summary that compares ClearDOM, Axe, and pa11y without requiring users to run the benchmark locally.

Concrete Backlog

Priority Work Item Why It Matters Acceptance Check
P0 Create a repo-local scan config or scan:src script. Prevents intentional fixtures and generated reports from making ClearDOM look broken when run against itself. pnpm start -- scan . or the documented dev command reports product-source health without fixture noise.
P0 Ignore generated benchmark report artifacts during scans. Generated HTML/Markdown reports under examples/wcag-benchmark/reports created many self-scan findings. Full-repo scan no longer includes examples/wcag-benchmark/reports/* unless explicitly targeted.
P1 Extract CLI parsing and command handlers. Reduces regression risk as commands keep growing. src/cli.ts becomes a thin router; existing CLI tests still pass unchanged.
P1 Consolidate report HTML. Keeps scan HTML and shareable report HTML visually and semantically aligned. One renderer serves both paths; tests cover both commands.
P1 Split test scripts by speed and dependency. Shortens local feedback while preserving browser coverage. pnpm test:fast runs source/CLI/rule tests; pnpm test:runtime runs Chromium tests.
P2 Generate command and rule docs from source metadata. The README is large and likely to drift as commands and rule metadata evolve. A docs generation check fails when command help, README command list, and rule catalog diverge.
P2 Audit design-system placeholder semantics. Current mappings can treat placeholder-like props as labels for custom text inputs. That may hide real label debt. Explicit tests decide the intended behavior for TextInput placeholder="Email" with no label prop.
P2 Improve HTML report interaction. Reports are useful, but larger scans need filtering and grouping to stay readable. Reports support grouped navigation and filters without losing static accessibility.

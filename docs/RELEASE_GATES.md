# ClearDOM 1.0 release gates

The canonical machine-readable thresholds live in `release-gates.json`; `pnpm release:gates -- --stage <alpha|beta|rc|final>` evaluates the evidence required for that stage. Alpha requires only the measured `contracts` fragment. Beta adds source/runtime/native conformance, GA4 configuration, and alpha history. RC and final require every precision, performance, fix-safety, security, platform, supply-chain, and history category. A missing required input is a failing gate, never an implicit pass.

Beta evidence is required per stack, including a unique, stack-owned runnable clean app, a unique broken app, a case manifest, source plus runtime/native completion, fix verification, and protection verification. Shared snippets and reused generic demo roots do not satisfy this gate. RC and final evidence additionally require at least 95% observed precision, 20 current reviewed automated findings, and five independently reviewed holdout findings for every default-blocking rule; an aggregate precision number cannot hide a weak or unsampled rule. Corpus entries pin a commit and license review, point to separately reviewed ground truth, declare `refinement` or `holdout`, and must run in shadow mode. Holdout projects cannot be used to tune rules.

Prereleases advance in order: alpha after contract and runner stability, beta after conformance, release candidate after corpus/security/performance evidence, and 1.0 only when every mandatory gate passes on the same commit.

The OSS corpus manifest records repository, full commit SHA, reviewed license source, adapter family, platform, and a narrow scan target. Ground truth is commit-bound and stored separately from scanner output. Unambiguous labels require a documented reviewer; disputed blocking labels require two independent reviewers. Corpus scans use sparse disposable checkouts, assert a clean Git status, and never apply fixes upstream. An empty corpus is a CI failure, not a successful no-op.

`pnpm test:corpus` writes a commit- and stage-bound precision fragment under `.cleardom/evidence/`. Resolved false-positive labels remain in ground truth as historical regression evidence; only findings whose current detection mode is `automated` contribute to precision. `pnpm corpus:discover` scans the pinned candidate paths and orders automated candidates by a stable SHA-256 selection hash within rule/platform groups. `pnpm evidence:assemble -- --stage <stage>` rejects missing required categories, mixed commits, mixed stages, duplicate values, and secret-like fields before writing `.cleardom/release-evidence.json`. The RC/final gate independently recomputes combined and holdout corpus counts and cross-checks the assembled values.

Source and rendered conformance jobs write their own fragments. Device, performance, fix-safety, security, platform, supply-chain, and release-history jobs record their measured JSON with `pnpm evidence:record -- <category> <values.json> --stage <stage>`; the command binds it to the current commit and stage and refuses categories owned by automated corpus/conformance/contract jobs. External attestations must contain results only, never credentials or the GA4 secret.

Alpha builds intentionally embed an empty GA4 secret. Beta, RC, and final builds fail unless `CLEARDOM_GA4_API_SECRET` is present and matches the release-secret format. The value is written only into the ephemeral build output, never into evidence or source history. Prepublish gates validate reproducibility, generated SBOM/checksums, and all same-commit measurements. After publication, `pnpm release:gates:postpublish` verifies npm provenance and the published SBOM/checksum assets without creating a circular prepublish dependency.
